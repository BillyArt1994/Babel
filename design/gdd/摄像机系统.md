# 摄像机系统

> **状态**: 设计中
> **作者**: 用户 + Claude
> **最后更新**: 2026-03-29
> **实现支柱**: 支柱 6（倒金字塔的绝望螺旋）、支柱 2（压力递增的绝望感）

## 概述

摄像机系统负责随通天塔的建造进度动态调整摄像机垂直高度。游戏运行期间，摄像机目标 Y 坐标始终跟踪当前活跃层的世界高度（由塔建造系统提供），并以平滑插值（Lerp）方式持续向目标位置靠近。摄像机不做水平移动，也无需感知鼠标或玩家操作。系统极简，但效果关键：随着塔一层层升高，玩家的视野随之上移，视觉上传递出"已无路可退"的压迫感，直接服务支柱 6。

## 玩家幻想

摄像机系统服务的不是"感知摄像机存在"，而是"感知塔在不断升高"。理想体验是：玩家专注消灭人类时，浑然不觉摄像机在动；但偶尔抬眼一看，发现屏幕中的世界悄悄变高了，地面变远了——这种不知不觉间"形势已然恶化"的感受，是支柱 2（压力递增的绝望感）和支柱 6（倒金字塔的绝望螺旋）的视觉载体。摄像机是"无声的叙事者"。

## 详细设计

### 核心规则

1. 摄像机为固定 Orthographic 模式，X/Z 坐标固定不变，仅 Y 坐标动态更新
2. 游戏运行期间（`GameLoopManager.IsPlaying() == true`），每帧更新：
   - `targetY = TowerBuildingSystem.GetActiveLayerHeight()`
   - `camera.position.y = Lerp(camera.position.y, targetY, CAMERA_LERP_SPEED × deltaTime)`
3. 游戏未运行（暂停、结算、主菜单）时，停止更新摄像机位置
4. 游戏开始时，摄像机 Y 直接设置为第 1 层高度（`layerHeight[1] = 0`），不做平滑过渡

### 状态与转换

| 状态 | 摄像机行为 | 进入条件 | 退出条件 |
|------|-----------|---------|---------|
| 游戏中 | 每帧 Lerp 向 `targetY` | `OnGameStart` 事件 | `OnGameEnd` 事件 |
| 暂停/结算 | 冻结在当前位置 | `OnGamePause` / `OnGameEnd` | `OnGameResume` / 新局开始 |
| 初始化 | 直接设置到第 1 层高度 | `OnGameStart` | 立即切入"游戏中"状态 |

### 与其他系统的交互

| 系统 | 方向 | 接口 |
|------|------|------|
| 塔建造系统 | 输入 ← | `GetActiveLayerHeight(): float` — 每帧查询目标 Y |
| 游戏循环管理器 | 输入 ← | 监听 `OnGameStart` 初始化；查询 `IsPlaying()` 决定是否更新 |

## 公式

1. **摄像机每帧目标 Y：**
   ```
   targetY = TowerBuildingSystem.GetActiveLayerHeight()
            = LAYER_BASE_HEIGHT + (activeLayer - 1) × LAYER_HEIGHT_STEP
            = 0 + (activeLayer - 1) × 3.0
   ```
   - 范围：[0, 27]（第 1 层 → 第 10 层，默认参数）

2. **Lerp 平滑移动（每帧执行）：**
   ```
   camera.y(t+Δt) = Lerp(camera.y(t), targetY, CAMERA_LERP_SPEED × Δt)
   ```
   - `CAMERA_LERP_SPEED` 建议值：`2.0`（每秒缩小约 86% 的差距）
   - Δt 典型值：0.016s（60fps）→ 每帧移动距离 ≈ `gap × 0.032`

3. **初始化（直接设置，非 Lerp）：**
   ```
   camera.y = layerHeight[1] = 0
   ```

## 边缘情况

1. **`GetActiveLayerHeight()` 在帧内多次返回不同值（层刚切换）**
   - 处理：每帧只查询一次 `targetY`，Lerp 自然收敛到新目标
   - 原因：Lerp 无状态跳变，天然平滑

2. **Lerp 永远无法精确到达 targetY（浮点收敛问题）**
   - 处理：若 `|camera.y - targetY| < SNAP_THRESHOLD`（如 0.01 Unity 单位），直接 `camera.y = targetY`
   - 原因：防止视觉上的"微颤"

3. **游戏暂停时摄像机仍在向 targetY 靠近**
   - 处理：暂停时跳出更新逻辑（`IsPlaying()` 检查），位置冻结
   - 原因：结算界面需要稳定画面

4. **多局游戏之间，上一局摄像机停在高层**
   - 处理：新局 `OnGameStart` 事件触发时，直接重置摄像机 Y = 0，无过渡
   - 原因：保证每局开始时玩家视角一致

## 依赖关系

**上游依赖（此系统依赖的系统）：**

| 系统 | 依赖类型 | 接口 |
|-----|---------|------|
| 塔建造系统 | 软依赖 | `GetActiveLayerHeight(): float` — 每帧查询摄像机目标 Y |
| 游戏循环管理器 | 软依赖 | `OnGameStart` 事件（初始化摄像机位置）；`IsPlaying()` 查询（控制更新开关） |

**下游依赖（依赖此系统的系统）：** 无（终端系统）

## 调优参数

| 参数名 | 默认值 | 安全范围 | 说明 |
|--------|--------|---------|------|
| `CAMERA_LERP_SPEED` | 2.0 | [0.5, 10.0] | Lerp 速度系数。低于 0.5 摄像机明显滞后；高于 10.0 接近瞬移，失去"缓慢上升"的压迫感 |
| `SNAP_THRESHOLD` | 0.01 | [0.001, 0.1] | 距离目标小于此值时直接 snap，防止浮点微颤 |
| `CAMERA_ORTHOGRAPHIC_SIZE` | 待定（约 9-10） | — | 视野大小，需调试到"第 1 层时可见约 5-6 层高度"。精确值依赖场景 Pixels Per Unit 比例，需在 Unity 中实测 |
| `CAMERA_X` | 固定（场景中轴线） | — | 摄像机水平位置，硬编码，不参与运行时计算 |

**相互影响：**
- `CAMERA_LERP_SPEED` 过低 → 摄像机明显落后于塔的实际进度，视觉上显得"脱节"
- `CAMERA_ORTHOGRAPHIC_SIZE` 过大 → 场景元素显得很小，后期层的威胁感下降

## 视觉/音频需求

无特殊需求。摄像机移动本身不需要音效或专属视觉效果。视觉压迫感来自场景内容（塔的高度、人类涌入），而非摄像机特效。

## UI 需求

无。摄像机系统不直接驱动任何 UI 元素。

## 验收标准

**功能测试：**
1. 游戏开始时，摄像机 Y 坐标正确初始化为 0（第 1 层高度），无异常位置
2. 游戏进行中，每帧 `camera.y` 向 `GetActiveLayerHeight()` 的方向平滑移动
3. 层从第 1 层切换至第 10 层过程中，摄像机 Y 最终稳定在对应高度（范围 [0, 27]）
4. `CAMERA_LERP_SPEED = 2.0` 时，视觉上摄像机移动流畅，无卡顿感
5. 游戏暂停时，摄像机位置在整个暂停期间保持不变
6. 多局连续进行时，新局开始摄像机立即回到 Y=0，不出现上局残留位置

**边缘情况测试：**
7. `|camera.y - targetY| < 0.01` 时摄像机精确 snap，无微颤

**性能标准：**
8. 摄像机 `Update()` 每帧耗时 ≤ 0.1ms

## 待解决问题

1. **`CAMERA_ORTHOGRAPHIC_SIZE` 精确值**：需在 Unity 场景搭建后实测，确认哪个值能在第 1 层时展示约 5-6 层的视野高度。当前标注为"待定"。
2. **层切换后 Lerp 追赶速度是否足够快**：当高层之间高度差为 3 Unity 单位，`CAMERA_LERP_SPEED = 2.0` 的追赶速度约需 1-2 秒到达新目标，是否有"摄像机跟不上"的体验问题，需实际测试确认。如有问题，可提高 `CAMERA_LERP_SPEED` 或改用非线性插值（如 SmoothDamp）。
