# 技能数据库

> **状态**: 已完成
> **作者**: 用户 + Claude
> **最后更新**: 2026-03-28
> **实现支柱**: 支柱 3（Build 的策略深度）、支柱 5（简洁的复杂性）

## 概述

技能数据库是游戏所有神力技能的配置仓库，以 Unity ScriptableObject 形式存在。每个技能是一个独立的 `SkillData` 资产，存储该技能的静态属性（伤害、冷却、范围、蓄力时间等），以及元数据（名称、描述、图标、类别）。

技能数据库本身不执行任何逻辑——它是纯数据层。技能系统读取这些数据来决定技能的行为；升级系统读取这些数据来生成升级选项；游戏 HUD 读取名称和图标来渲染 UI。数据与逻辑的分离使得设计师可以在不修改代码的情况下调整技能数值。

## 玩家幻想

技能数据库对玩家不可见，但它是"Build 策略深度"体验的基础。当玩家在升级界面看到"天降陨石：范围 3m，伤害 150，冷却 2.5s"与"神雷锁链：跳跃 3 次，每跳伤害 80，冷却 1.5s"时，他们做出的是一个有信息支撑的战略决策——这种决策感来源于数据库中清晰定义的数值。

这是一个"你不爱用，但你会因它消失而崩溃"的系统。它让技能系统可以简单地读取数据而非硬编码，让设计师可以通过 Inspector 调整数值而无需改代码。

## 详细设计

### 核心规则

每个技能以一个 `SkillData` ScriptableObject 资产存在，保存在 `Assets/Data/Skills/` 目录下。

**`SkillData` 字段定义：**

#### 元数据字段
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `skillId` | `string` | 唯一标识符（如 `"divine_finger"`），用于代码引用 |
| `skillName` | `string` | 显示名称（如"神罚之指"） |
| `description` | `string` | 升级界面显示的描述文字 |
| `icon` | `Sprite` | 升级界面和 HUD 显示的图标 |
| `skillType` | `SkillType` (enum) | `ClickForm`（点击形态）或 `Passive`（被动增强） |
| `passiveCategory` | `PassiveCategory` (enum) | 仅 Passive 有效：`Frequency` / `Power` / `Effect` / `AutoAttack` |

#### 战斗数值字段
| 字段名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `damage` | `float` | `0` | 每次命中基础伤害（0 表示无直接伤害，如时间凝滞） |
| `cooldown` | `float` | `1.0` | 技能冷却时间（秒），ClickForm 专用 |
| `chargeTime` | `float` | `0` | 最大蓄力时间（秒），0 表示无需蓄力 |
| `aoeRadius` | `float` | `0` | AOE 范围半径（Unity 单位），0 表示单体技能 |
| `piercing` | `bool` | `false` | 是否穿透敌人（圣光洪流用） |
| `chainCount` | `int` | `0` | 跳跃次数（神雷锁链用），0 表示不跳跃 |
| `slowPercent` | `float` | `0` | 减速百分比 [0, 1]（时间凝滞用），0 表示不减速 |
| `slowDuration` | `float` | `0` | 减速持续时间（秒） |
| `triggerInterval` | `float` | `0` | 自动触发间隔（秒），用于天雷 / 神火 / 追踪光球 |
| `spreadOnKill` | `bool` | `false` | 击杀时传播效果（瘟疫用） |
| `maxStackCount` | `int` | `0` | 可叠加层数上限（神罚印记用），0 表示不可叠加 |
| `bonusDamagePerStack` | `float` | `0` | 每层叠加的伤害加成（神罚印记用） |

#### 升级系统字段
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `isStarterSkill` | `bool` | 是否为起始可选技能（游戏开始时 3 选 1） |
| `upgradesFrom` | `SkillData` | 此技能是哪个技能的进化形态（null 表示初始技能） |
| `weight` | `float` | 升级池中出现的权重（越高越频繁出现），默认 1.0 |

### 状态与转换

数据库系统无运行时状态——`SkillData` 资产在游戏启动时加载到内存，全生命周期只读。无状态转换。

### 与其他系统的交互

**提供给其他系统的数据（只读）：**

| 消费方 | 读取的字段 | 用途 |
|--------|-----------|------|
| 技能系统 | 全部战斗数值字段 | 执行技能效果（伤害、范围、冷却等） |
| 升级系统 | `skillType`、`upgradesFrom`、`weight`、`isStarterSkill` | 生成升级选项池、起始技能选择 |
| 游戏 HUD | `skillName`、`icon`、`cooldown` | 显示当前技能名、图标、冷却进度 |
| 升级选择 UI | `skillName`、`description`、`icon` | 渲染 3 选 1 卡片 |

**输入依赖：** 无（数据库不依赖任何运行时系统）。

## 公式

技能数据库本身不执行计算，但以下是各字段的预期使用方式，供技能系统参考：

1. **实际伤害（技能系统执行）**
   ```
   actualDamage = skillData.damage × damageMultiplier
   ```
   - `damageMultiplier` 由被动技能叠加（如"神力" +30%），由技能系统计算

2. **实际冷却（技能系统执行）**
   ```
   actualCooldown = skillData.cooldown × (1 - cooldownReduction)
   ```
   - `cooldownReduction` 由"神速"等被动提供（0.0 - 0.8 范围，防止冷却归零）

3. **蓄力伤害系数（输入系统 + 技能系统协作）**
   ```
   chargeRatio = clamp(holdDuration / skillData.chargeTime, 0, 1)
   ```
   - `holdDuration` 由输入系统提供
   - 如何利用 `chargeRatio` 由技能系统自行定义（如线性伤害放大）

4. **升级权重抽取（升级系统执行）**
   ```
   P(skill) = skillData.weight / Σ(weight of all eligible skills)
   ```

## 边缘情况

1. **`skillId` 重复**
   - 情况：两个 SkillData 资产拥有相同的 `skillId`
   - 处理：在 `OnValidate()` 中扫描所有 SkillData 资产，检测重复 ID，在 Inspector 中显示警告
   - 原因：技能系统通过 ID 查找技能，重复 ID 会导致不可预测行为

2. **`upgradesFrom` 循环引用**
   - 情况：技能 A 的 `upgradesFrom` 指向技能 B，技能 B 的 `upgradesFrom` 又指向技能 A
   - 处理：`OnValidate()` 检测循环引用，警告并清空引用
   - 原因：升级系统遍历进化链时会无限循环

3. **`chargeTime = 0` 但技能系统期望蓄力**
   - 情况：技能系统错误地对一个非蓄力技能读取 `chargeTime`
   - 处理：`chargeTime = 0` 定义为"无蓄力"，技能系统应在使用前检查 `chargeTime > 0`
   - 原因：防止除以零（`chargeRatio = holdDuration / chargeTime`）

4. **`slowPercent` 超出 [0, 1] 范围**
   - 情况：设计师误填 `slowPercent = 1.5`（150% 减速）
   - 处理：`OnValidate()` 自动钳制到 [0, 0.99]（不允许完全停止）
   - 原因：100% 减速在游戏逻辑上等同于定身，应由单独的"定身"状态处理

5. **`weight = 0`**
   - 情况：设计师将某技能权重设为 0（永不出现）
   - 处理：升级系统跳过 weight ≤ 0 的技能，允许此行为（可用于临时禁用技能）
   - 原因：提供快速禁用技能的机制，无需删除资产

6. **图标为空（`icon = null`）**
   - 情况：SkillData 没有配置图标
   - 处理：UI 显示默认占位图标，不报错
   - 原因：开发阶段图标尚未制作时，不应影响游戏功能

## 依赖关系

**上游依赖（此系统依赖的系统）：** 无（纯数据层，不依赖任何运行时系统）

**下游依赖（依赖此系统的系统）：**

| 系统 | 依赖类型 | 读取的接口 |
|-----|---------|----------|
| 技能系统 | 硬依赖 | 所有战斗数值字段 |
| 升级系统 | 硬依赖 | `skillType`、`upgradesFrom`、`weight`、`isStarterSkill` |
| 游戏 HUD | 软依赖 | `skillName`、`icon`、`cooldown` |
| 升级选择 UI | 软依赖 | `skillName`、`description`、`icon` |

## 调优参数

所有战斗数值字段均可通过 Unity Inspector 直接调整，无需代码修改。推荐安全范围：

| 字段名 | 推荐范围 | 极端行为 |
|--------|---------|---------|
| `damage` | [0, 500] | > 500 可能单次点击秒杀精英单位，破坏节奏感 |
| `cooldown` | [0.3, 5.0] | < 0.3s 视觉反馈堆叠混乱；> 5s 玩家感觉技能"不存在" |
| `chargeTime` | [0, 3.0] | > 3s 蓄力过程太长，违反支柱 1 的节奏感 |
| `aoeRadius` | [0, 8.0] | > 8 可能覆盖整个屏幕，使点击位置失去意义 |
| `chainCount` | [0, 6] | > 6 跳跃目标可能不够，产生"链式找不到目标"的错误 |
| `slowPercent` | [0, 0.99] | 由 `OnValidate()` 自动钳制 |
| `weight` | [0, 5.0] | > 5 可能使某技能几乎必然出现，破坏 Build 多样性 |

## 验收标准

1. 每个 SkillData 资产可在 Unity Inspector 中独立创建和配置
2. 重复 `skillId` 在 Inspector 中显示警告（`OnValidate()`）
3. `upgradesFrom` 循环引用被检测并警告
4. `slowPercent` 超出范围时被自动钳制到 [0, 0.99]
5. 技能系统可通过 `skillId` 正确查找并读取任意 SkillData
6. 升级系统可正确读取 `weight` 字段生成加权随机升级选项
7. `icon = null` 时，UI 显示占位图而不崩溃
8. 修改任意 SkillData 字段后，游戏重新进入时生效（无需重编译代码）
