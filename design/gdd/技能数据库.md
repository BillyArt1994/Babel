# 技能数据库

> **状态**: 重构中
> **作者**: 用户 + Claude
> **最后更新**: 2026-04-17
> **实现支柱**: 支柱 3（Build 的策略深度）、支柱 5（简洁的复杂性）

## 概述

技能数据库是游戏所有神力技能的配置仓库。重构后改为 **CSV 驱动**架构：每个技能定义在一行 CSV 记录中，由 `SkillParser` 在游戏启动时解析为 `SkillConfig` 运行时对象，并注册到 `SkillDatabase` 字典供全局查询。

新架构的核心思路是 **Skill = Trigger + Effects**：每个技能由一个触发条件（Trigger）和 **1~3 个执行效果（Effect）** 组合而成（参考 Death Must Die 的多效果 Boon 设计）。CSV 中每个参数都是独立列——触发器参数（cooldown、chargeTime 等）和效果参数（damage、radius 等）各占一列，运行时按列名直接读取。这一设计使得设计师可在 Excel 中对任意参数列进行排序、筛选和批量修改。

> **多效果规则（参考 DMD）**：95% 的技能只使用 1 个 Effect，少数进化/传奇技能使用 2~3 个。同一 Trigger 触发时所有 Effect **同时执行**，共享同一个不可变的 `EffectContext` 快照。Effect 之间不能互相依赖执行结果。

技能数据库本身不执行任何逻辑——它是纯数据层。技能系统读取这些数据来决定技能的行为；升级系统读取这些数据来生成升级选项；游戏 HUD 读取名称和图标路径来渲染 UI。数据与逻辑的分离使得设计师可以在不修改代码的情况下调整技能数值。

## 玩家幻想

技能数据库对玩家不可见，但它是"Build 策略深度"体验的基础。当玩家在升级界面看到"天降陨石：范围 3m，伤害 150，冷却 2.5s"与"天雷：每 3 秒自动落雷，伤害 80"时，他们做出的是一个有信息支撑的战略决策——这种决策感来源于数据库中清晰定义的数值。

这是一个"你不爱用，但你会因它消失而崩溃"的系统。它让技能系统可以简单地读取数据而非硬编码，让设计师可以通过修改 CSV 调整数值而无需改代码。

## 详细设计

### 核心架构：Trigger + Effect

每个技能由两部分组合定义：

- **Trigger（触发条件）**：决定技能在何种条件下触发。支持类型：
  - `OnClick`：玩家点击屏幕时触发（可选蓄力，`chargeTime > 0`）
  - `OnHit`：命中目标时触发
  - `OnTimer`：按固定间隔自动触发
  - `OnKill`：击杀敌人时触发

- **Effect（执行效果）**：决定技能触发后产生的具体行为形态（参考 Death Must Die 的 Effect 分类设计）。支持类型：
  - `hit_single`：单体伤害（指定位置命中单个目标）
  - `hit_aoe`：范围伤害（以指定位置为圆心，radius 为半径的 AOE）
  - `hit_chain`：连锁伤害（闪电链式，从目标向周围跳跃）
  - `dot_aoe`：持续范围伤害（地面效果，每秒 tick）
  - `stat_buff`：属性修改（修改玩家属性：伤害倍率、冷却缩减、暴击率等）
  - `spawn_projectile`：生成投射物（追踪光球等自主移动实体）
  - `apply_status`：施加状态（给目标附加标记/减速/感染等）
  - `execute`：处决（对低于阈值生命值的目标造成致命伤害）

Trigger 与 Effect 的参数在 CSV 中**各自占据独立列**（如 `cooldown`、`damage`、`radius` 等），由 `SkillParser` 按列名直接读取为具体数值。不同类型只使用各自需要的列，其余列留空。

> **设计原则**：每个可调数值都是独立列，使设计师能在 Excel/WPS 中直接对任意参数列进行排序、筛选和批量修改，无需解析嵌套字符串。

### SkillConfig（运行时数据结构）

`SkillConfig` 是 CSV 一行数据对应的运行时纯数据结构，字段如下：

#### 元数据字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `skillId` | `string` | 唯一标识符（如 `"divine_finger"`），用于代码引用 |
| `skillName` | `string` | 显示名称（如"神罚之指"） |
| `description` | `string` | 升级界面显示的描述文字 |
| `iconPath` | `string` | 图标资源路径（如 `"Icons/divine_finger"`），由 UI 层异步加载 |

#### Trigger 字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `triggerType` | `string` | 触发器类型名（`OnClick` / `OnHit` / `OnTimer` / `OnKill`） |
| `cooldown` | `float` | 触发冷却时间（秒）。`OnClick` 必填，其他类型可为空 |
| `chargeTime` | `float` | 蓄力时间（秒）。仅 `OnClick` 使用，`> 0` 时启用蓄力机制 |
| `interval` | `float` | 自动触发间隔（秒）。仅 `OnTimer` 使用 |
| `chance` | `float` | 触发概率 [0, 1]。`OnHit`、`OnKill` 使用，`1.0` = 必定触发 |

#### Effect 字段

每个 Effect 槽位由 1 个类型列 + 7 个参数列组成。主效果不带前缀，第二/三效果分别使用 `e2`/`e3` 前缀。

**主效果（Effect 1）：**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `effectType` | `string` | 效果类型名（`hit_single` / `hit_aoe` / `hit_chain` / `dot_aoe` / `stat_buff` / `spawn_projectile` / `apply_status` / `execute`） |
| `damage` | `float` | 基础伤害值。`hit_single`、`hit_aoe`、`hit_chain` 使用 |
| `damageRatio` | `float` | 伤害比例系数（如 0.3 = 主伤害的 30%）。用于按比例伤害的效果 |
| `radius` | `float` | AOE 半径。`hit_aoe`、`dot_aoe`、`hit_chain` 使用 |
| `dps` | `float` | 每秒伤害。仅 `dot_aoe` 使用 |
| `duration` | `float` | 持续时间（秒）。`dot_aoe`、`stat_buff` 使用（`-1` = 永久） |
| `statName` | `string` | 目标属性名。仅 `stat_buff` 使用（如 `cooldownMult`、`damageMult`） |
| `statValue` | `float` | 属性修正量。仅 `stat_buff` 使用（正数=增益，负数=减益） |

**第二效果（Effect 2，`e2` 前缀）：**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `effect2Type` | `string` | 第二效果类型名（可为空，空 = 无第二效果） |
| `e2Damage` | `float` | 同 `damage`（第二效果使用） |
| `e2DamageRatio` | `float` | 同 `damageRatio` |
| `e2Radius` | `float` | 同 `radius` |
| `e2Dps` | `float` | 同 `dps` |
| `e2Duration` | `float` | 同 `duration` |
| `e2StatName` | `string` | 同 `statName` |
| `e2StatValue` | `float` | 同 `statValue` |

**第三效果（Effect 3，`e3` 前缀）：** 列名与 Effect 2 结构相同，前缀改为 `e3`。

> **可扩展性**：当引入新效果类型（如 `hit_chain`、`spawn_projectile`）时，可在对应位置追加新参数列（如 `chainCount`、`prefabPath`）。`SkillParser` 按列名匹配，新增列不影响已有数据。

> **多效果约束规则：**
> 1. Effect 槽位必须**连续填充**：不允许 `effect2Type` 为空而 `effect3Type` 有值
> 2. 所有 Effect 共享同一个**不可变** `EffectContext` 快照，不能互相依赖执行结果
> 3. 执行顺序固定为 `effect1 → effect2 → effect3`，但设计时不应依赖执行顺序
> 4. 同一技能的多个 `stat_buff` Effect 在同一次触发中**不会互相影响**（如 +伤害 和 -防御各自独立生效）
> 5. 未知或无效的子效果类型导致**整个技能**在解析阶段被拒绝（Fail-fast 原则）

#### 升级系统字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `level` | `int` | 技能等级（初始为 1，进化后递增） |
| `weight` | `float` | 升级池中出现的权重（越高越频繁出现），默认 `1.0` |
| `isStarterSkill` | `bool` | 是否为起始可选技能（游戏开始时 3 选 1） |
| `upgradesFrom` | `string` | 进化来源技能的 `skillId`（空字符串表示初始技能） |

### SkillParser

`SkillParser` 负责将 CSV `TextAsset` 解析为 `SkillConfig` 列表，流程如下：

1. 读取首行表头，建立**列名 → 列索引映射**（按列名匹配，而非固定列序号）
2. 逐行读取 CSV 数据行，按列索引提取各字段，处理空值与类型转换（`string` → `int` / `float` / `bool`）
3. 按 `triggerType` 字符串通过反射实例化对应的 Trigger 类，从对应列（`cooldown`、`chargeTime`、`interval`、`chance`）填充参数
4. 依次检查 `effectType`、`effect2Type`、`effect3Type`，非空的类型通过反射实例化对应的 Effect 类，从对应参数列填充参数，收集为 `List<IEffect>`
5. **连续性校验**：`effect2Type` 为空而 `effect3Type` 有值时，发出警告并忽略 `effect3`
6. **子效果有效性**：任一 Effect 类型无法识别时，拒绝整行并记录警告（Fail-fast）
7. 将解析结果注册到 `SkillDatabase`

**参数读取规则：**
- 每个参数占独立列，空单元格视为"该参数不适用"，使用该参数类型的默认值
- `SkillParser` 按列名（而非列序号）匹配字段，因此新增列不影响已有解析逻辑（前向兼容）
- 未知列名被静默忽略（不报错）

### SkillDatabase

`SkillDatabase` 是游戏全局单例，持有所有技能的运行时字典，提供以下查询接口：

| 方法签名 | 说明 |
|---------|------|
| `GetById(string skillId)` | 按唯一 ID 查找技能，不存在时返回 `null` |
| `GetAll()` | 返回所有技能列表 |
| `GetByTriggerType(string triggerType)` | 按触发器类型筛选技能列表 |
| `GetStarterSkills()` | 返回所有 `isStarterSkill = true` 的技能 |
| `GetUpgradesOf(string skillId)` | 返回所有以此技能为 `upgradesFrom` 的进化形态 |

`SkillDatabase` 本身不读取 CSV，由 `SkillParser` 在游戏启动时负责填充。填充完成后数据库全生命周期只读。

### 与其他系统的交互

**提供给其他系统的数据（只读）：**

| 消费方 | 读取的字段 | 用途 |
|--------|-----------|------|
| 技能系统 | `triggerType`、trigger 参数列、`effectType`/`effect2Type`/`effect3Type`、各效果参数列 | 实例化 Trigger / Effect（多 Effect 时组装 CompositeEffect），执行技能逻辑 |
| 升级系统 | `upgradesFrom`、`weight`、`isStarterSkill`、`level` | 生成升级选项池、起始技能选择、进化链 |
| 游戏 HUD | `skillName`、`iconPath`、`cooldown` | 显示当前技能名、图标、冷却进度 |
| 升级选择 UI | `skillName`、`description`、`iconPath` | 渲染 3 选 1 卡片 |

**输入依赖：** 无（数据库不依赖任何运行时系统）。`SkillParser` 依赖 Unity `Resources` 或 `Addressables` 加载 CSV `TextAsset`。

## 公式

技能数据库本身不执行计算，以下是各参数的预期使用方式，供技能系统参考：

1. **实际伤害（技能系统从 `damage` 列读取后执行）**
   ```
   actualDamage = config.damage × damageMultiplier
   ```
   - `damageMultiplier` 由被动技能叠加（如"神力" +30%），由技能系统计算

2. **实际冷却（技能系统从 `cooldown` 列读取后执行）**
   ```
   actualCooldown = config.cooldown × (1 - cooldownReduction)
   ```
   - `cooldownReduction` 由"神速"等被动提供（0.0 - 0.8 范围，防止冷却归零）

3. **蓄力伤害系数（输入系统 + 技能系统协作）**
   ```
   chargeRatio = clamp(holdDuration / config.chargeTime, 0, 1)
   ```
   - `holdDuration` 由输入系统提供
   - 仅 `OnClick` 且 `chargeTime > 0` 时执行此逻辑
   - 如何利用 `chargeRatio` 由技能系统自行定义（如线性伤害放大）

4. **自动触发间隔（OnTimer 触发器）**
   ```
   triggerInterval = config.interval
   ```

5. **升级权重抽取（升级系统执行）**
   ```
   P(skill) = skillConfig.weight / Σ(weight of all eligible skills)
   ```

## 边缘情况

1. **`skillId` 重复**
   - 情况：CSV 中两行拥有相同的 `skillId`
   - 处理：`SkillParser` 解析时检测重复 ID，后出现的记录覆盖前者并输出警告日志
   - 原因：`SkillDatabase` 为 `Dictionary<string, SkillConfig>`，重复键会静默覆盖，需显式告警

2. **`upgradesFrom` 引用不存在的 skillId**
   - 情况：某技能的 `upgradesFrom` 填写了一个 CSV 中不存在的 ID
   - 处理：`SkillParser` 完成全量加载后进行引用完整性校验，不存在的引用置为空字符串并输出警告
   - 原因：升级系统遍历进化链时会查询 `GetById()`，返回 `null` 会导致空引用异常

3. **`upgradesFrom` 循环引用**
   - 情况：技能 A 的 `upgradesFrom` 指向技能 B，技能 B 的 `upgradesFrom` 又指向技能 A
   - 处理：`SkillDatabase.GetUpgradesOf()` 遍历时设置最大深度上限（如 10），超出则截断并报错
   - 原因：升级系统遍历进化链时会无限循环

4. **必需参数列为空**
   - 情况：`OnClick` 触发器的 `cooldown` 列为空
   - 处理：对应 Trigger / Effect 类在实例化时使用预定义默认值，并输出警告日志
   - 原因：防止参数缺失导致运行时异常，同时让数据错误可追溯

5. **`chargeTime` 参数为 0 但技能系统计算蓄力比例**
   - 情况：技能系统对 `chargeTime = 0` 的技能执行除法
   - 处理：技能系统在读取 `chargeTime` 后检查 `> 0`，为 0 时 `chargeRatio` 直接返回 1.0
   - 原因：防止除以零（`chargeRatio = holdDuration / chargeTime`）

6. **`weight = 0`**
   - 情况：设计师将某技能权重设为 0（永不出现）
   - 处理：升级系统跳过 `weight ≤ 0` 的技能，允许此行为（可用于临时禁用技能）
   - 原因：提供快速禁用技能的机制，无需删除 CSV 行

7. **`iconPath` 对应资源不存在**
   - 情况：`iconPath` 路径填写错误或图标尚未制作
   - 处理：UI 层加载失败时显示默认占位图标，不报错不崩溃
   - 原因：开发阶段图标尚未制作时，不应影响游戏功能

8. **CSV 文件格式错误（列数不符）**
   - 情况：某行列数与表头不匹配（预期 37 列）
   - 处理：`SkillParser` 跳过该行并输出包含行号的警告日志，继续解析后续行
   - 原因：单行错误不应阻断整个数据库的加载

9. **`effect2Type` 为空但 `effect3Type` 有值（槽位不连续）**
   - 情况：设计师跳过第二效果直接填写第三效果
   - 处理：`SkillParser` 输出警告日志，忽略 `effect3`，仅保留主效果
   - 原因：多效果必须连续填充，跳跃填充是数据错误

10. **`effect2Type` 类型未知（子效果无法识别）**
    - 情况：`effect2Type` 填写了不存在的效果类型字符串
    - 处理：`SkillParser` 拒绝整行技能数据并输出错误日志（Fail-fast 原则）
    - 原因：部分有效的多效果技能可能产生意外行为，优先保证配置完整性

11. **第二效果参数列有值但 `effect2Type` 为空**
    - 情况：`e2Damage`、`e2Radius` 等列有值但 `effect2Type` 为空
    - 处理：`SkillParser` 忽略孤立参数，输出警告日志
    - 原因：无类型的参数无法实例化 Effect，视为数据填写错误

12. **多效果中的同触发 stat_buff 互影响**
    - 情况：同一技能同时包含 `stat_buff`（+伤害）和 `hit_single`（伤害），stat_buff 可能影响同次触发的 hit_single
    - 处理：所有 Effect 共享同一个不可变 `EffectContext` 快照，同一触发内的 stat_buff **不会**影响同触发的其他兄弟 Effect
    - 原因：防止效果执行顺序影响最终结果（顺序无关性原则）

## 调优参数

所有战斗数值均存于 CSV 的独立列中，修改后重启游戏即生效，无需重编译代码。推荐安全范围：

| 参数名 | CSV 列名 | 推荐范围 | 极端行为 |
|--------|---------|---------|---------|
| 伤害 | `damage` | [0, 500] | > 500 可能单次点击秒杀精英单位，破坏节奏感 |
| 冷却 | `cooldown` | [0.3, 5.0] | < 0.3s 视觉反馈堆叠混乱；> 5s 玩家感觉技能"不存在" |
| 蓄力 | `chargeTime` | [0, 3.0] | > 3s 蓄力过程太长，违反支柱 1 的节奏感 |
| AOE 半径 | `radius` | [0, 8.0] | > 8 可能覆盖整个屏幕，使点击位置失去意义 |
| 自动间隔 | `interval` | [1.0, 10.0] | < 1s 自动攻击频率过高，噪音感强；> 10s 存在感太低 |
| 升级权重 | `weight` | [0, 5.0] | > 5 可能使某技能几乎必然出现，破坏 Build 多样性 |
| 触发概率 | `chance` | [0.0, 1.0] | 固定 1.0 表示必然触发；< 0.1 体感几乎不触发 |

## 验收标准

1. `SkillParser` 能正确解析标准格式的 `skills.csv`（37 列），不抛出异常
2. 解析完成后 `SkillDatabase.GetById("divine_finger")` 返回正确的 `SkillConfig` 对象
3. `triggerType` 和 `effectType` 字符串能通过反射正确实例化对应的 Trigger / Effect 类
4. 各参数列（如 `damage`、`cooldown`、`radius`）的值能被正确读取并供 Trigger / Effect 类使用
5. 重复 `skillId` 时输出警告日志，后出现的记录覆盖前者，不崩溃
6. `upgradesFrom` 引用不存在的 ID 时输出警告日志，置空，不崩溃
7. CSV 某行列数不符（非 37 列）时跳过该行并输出行号警告，继续加载后续行
8. `GetStarterSkills()` 正确返回所有 `isStarterSkill = true` 的技能
9. 升级系统可正确读取 `weight` 字段生成加权随机升级选项
10. `iconPath` 对应资源不存在时，UI 显示占位图而不崩溃
11. 修改 CSV 数值后，游戏重新启动时生效（无需重编译代码）
12. **多效果技能**（如 `meteor_evolved`）解析后 `SkillConfig` 包含正确数量的 Effect 配置
13. **effect2/3 为空的技能**解析后只包含 1 个 Effect，不产生空指针或多余实例
14. **effect2 为空但 effect3 有值**的行触发警告日志并忽略 effect3
15. **未知 effect2Type**导致整行技能被拒绝，不部分加载
16. **空参数列处理**：空单元格按类型默认值填充（float=0, string=""），不报错
