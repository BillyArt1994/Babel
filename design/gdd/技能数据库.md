# 技能数据库

> **状态**: 重构中
> **作者**: 用户 + Claude
> **最后更新**: 2026-04-17
> **实现支柱**: 支柱 3（Build 的策略深度）、支柱 5（简洁的复杂性）

## 概述

技能数据库是游戏所有神力技能的配置仓库。重构后改为 **CSV 驱动**架构：每个技能定义在一行 CSV 记录中，由 `SkillParser` 在游戏启动时解析为 `SkillConfig` 运行时对象，并注册到 `SkillDatabase` 字典供全局查询。

新架构的核心思路是 **Skill = Trigger + Effect**：每个技能由一个触发条件（Trigger）和一个执行效果（Effect）组合而成。CSV 中以字符串形式记录触发器类型与参数、效果类型与参数，运行时通过反射实例化对应的具体类。这一设计使得添加新技能只需增加 CSV 行，无需修改代码。

技能数据库本身不执行任何逻辑——它是纯数据层。技能系统读取这些数据来决定技能的行为；升级系统读取这些数据来生成升级选项；游戏 HUD 读取名称和图标路径来渲染 UI。数据与逻辑的分离使得设计师可以在不修改代码的情况下调整技能数值。

## 玩家幻想

技能数据库对玩家不可见，但它是"Build 策略深度"体验的基础。当玩家在升级界面看到"天降陨石：范围 3m，伤害 150，冷却 2.5s"与"天雷：每 3 秒自动落雷，伤害 80"时，他们做出的是一个有信息支撑的战略决策——这种决策感来源于数据库中清晰定义的数值。

这是一个"你不爱用，但你会因它消失而崩溃"的系统。它让技能系统可以简单地读取数据而非硬编码，让设计师可以通过修改 CSV 调整数值而无需改代码。

## 详细设计

### 核心架构：Trigger + Effect

每个技能由两部分组合定义：

- **Trigger（触发条件）**：决定技能在何种条件下触发。支持类型：
  - `OnClick`：玩家点击屏幕时触发（可选蓄力，`chargeTime > 0`）
  - `OnHit`：命中目标时触发
  - `OnTimer`：按固定间隔自动触发
  - `OnKill`：击杀敌人时触发

- **Effect（执行效果）**：决定技能触发后产生的效果。支持类型：
  - `Damage`：造成伤害（支持 AOE 范围参数）
  - `Dot`：持续伤害（Damage over Time）
  - `Buff`：施加增益或减益状态
  - `Summon`：召唤单位或特效

Trigger 与 Effect 各自携带一组键值对参数（格式：`key=value;key=value`），由 `SkillParser` 在解析时转换为具体数值。

---

### SkillConfig（运行时数据结构）

`SkillConfig` 是 CSV 一行数据对应的运行时纯数据结构，字段如下：

#### 元数据字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `skillId` | `string` | 唯一标识符（如 `"divine_finger"`），用于代码引用 |
| `skillName` | `string` | 显示名称（如"神罚之指"） |
| `description` | `string` | 升级界面显示的描述文字 |
| `iconPath` | `string` | 图标资源路径（如 `"Icons/divine_finger"`），由 UI 层异步加载 |

#### Trigger 字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `triggerType` | `string` | 触发器类型名（`OnClick` / `OnHit` / `OnTimer` / `OnKill`） |
| `triggerParams` | `string` | 触发器参数，`key=value;key=value` 格式（如 `cooldown=1.0;chargeTime=0.5`） |

#### Effect 字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `effectType` | `string` | 效果类型名（`Damage` / `Dot` / `Buff` / `Summon`） |
| `effectParams` | `string` | 效果参数，`key=value;key=value` 格式（如 `damage=50;radius=0`） |

#### 升级系统字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `level` | `int` | 技能等级（初始为 1，进化后递增） |
| `weight` | `float` | 升级池中出现的权重（越高越频繁出现），默认 `1.0` |
| `isStarterSkill` | `bool` | 是否为起始可选技能（游戏开始时 3 选 1） |
| `upgradesFrom` | `string` | 进化来源技能的 `skillId`（空字符串表示初始技能） |

---

### SkillParser

`SkillParser` 负责将 CSV `TextAsset` 解析为 `SkillConfig` 列表，流程如下：

1. 逐行读取 CSV，跳过首行表头
2. 按列顺序提取各字段，处理空值与类型转换（`string` → `int` / `float` / `bool`）
3. 对 `triggerParams` 和 `effectParams` 拆分为 `Dictionary<string, string>` 备用
4. 通过反射按 `triggerType` / `effectType` 字符串实例化对应的 Trigger / Effect 类
5. 将解析结果注册到 `SkillDatabase`

**参数解析规则：**
- 格式：`key=value;key=value`（分号分隔，等号连接）
- 空参数字段（`triggerParams` 或 `effectParams` 为空字符串）视为无参数，不报错

---

### SkillDatabase

`SkillDatabase` 是游戏全局单例，持有所有技能的运行时字典，提供以下查询接口：

| 方法签名 | 说明 |
|---------|------|
| `GetById(string skillId)` | 按唯一 ID 查找技能，不存在时返回 `null` |
| `GetAll()` | 返回所有技能列表 |
| `GetByTriggerType(string triggerType)` | 按触发器类型筛选技能列表 |
| `GetStarterSkills()` | 返回所有 `isStarterSkill = true` 的技能 |
| `GetUpgradesOf(string skillId)` | 返回所有以此技能为 `upgradesFrom` 的进化形态 |

`SkillDatabase` 本身不读取 CSV，由 `SkillParser` 在游戏启动时负责填充。填充完成后数据库全生命周期只读。

### CSV 格式定义

文件位置：`Assets/Data/Skills/skills.csv`

```
skillId,skillName,description,iconPath,triggerType,triggerParams,effectType,effectParams,level,weight,isStarterSkill,upgradesFrom
divine_finger,神罚之指,天神之怒降临指尖,Icons/divine_finger,OnClick,cooldown=1.0;chargeTime=0.5,Damage,damage=50;radius=0,1,1.0,true,
meteor,天降陨石,陨石从天而降,Icons/meteor,OnClick,cooldown=2.5;chargeTime=1.0,Damage,damage=150;radius=3.0,1,1.0,true,
thunder_auto,天雷,自动降下雷击,Icons/thunder,OnTimer,interval=3.0,Damage,damage=80;radius=0,1,0.8,false,
plague,瘟疫,击杀传播死亡,Icons/plague,OnKill,chance=1.0,Damage,damage=20;radius=2.0,1,0.6,false,
```

**列顺序（固定）：**

| # | 列名 | 说明 |
|---|-----|------|
| 1 | `skillId` | 唯一 ID |
| 2 | `skillName` | 显示名称 |
| 3 | `description` | 描述文字 |
| 4 | `iconPath` | 图标路径 |
| 5 | `triggerType` | 触发器类型 |
| 6 | `triggerParams` | 触发器参数（可为空） |
| 7 | `effectType` | 效果类型 |
| 8 | `effectParams` | 效果参数（可为空） |
| 9 | `level` | 技能等级 |
| 10 | `weight` | 升级权重 |
| 11 | `isStarterSkill` | 是否起始技能（`true`/`false`） |
| 12 | `upgradesFrom` | 进化来源 ID（可为空） |

### 状态与转换

数据库系统无运行时状态——`SkillConfig` 对象在游戏启动时由 `SkillParser` 填充到 `SkillDatabase`，全生命周期只读。无状态转换。

### 与其他系统的交互

**提供给其他系统的数据（只读）：**

| 消费方 | 读取的字段 | 用途 |
|--------|-----------|------|
| 技能系统 | `triggerType`、`triggerParams`、`effectType`、`effectParams` | 实例化 Trigger / Effect，执行技能逻辑 |
| 升级系统 | `upgradesFrom`、`weight`、`isStarterSkill`、`level` | 生成升级选项池、起始技能选择、进化链 |
| 游戏 HUD | `skillName`、`iconPath`、`triggerParams.cooldown` | 显示当前技能名、图标、冷却进度 |
| 升级选择 UI | `skillName`、`description`、`iconPath` | 渲染 3 选 1 卡片 |

**输入依赖：** 无（数据库不依赖任何运行时系统）。`SkillParser` 依赖 Unity `Resources` 或 `Addressables` 加载 CSV `TextAsset`。

## 公式

技能数据库本身不执行计算，以下是各参数的预期使用方式，供技能系统参考：

1. **实际伤害（技能系统从 effectParams 读取后执行）**
   ```
   actualDamage = effectParams["damage"] × damageMultiplier
   ```
   - `damageMultiplier` 由被动技能叠加（如"神力" +30%），由技能系统计算

2. **实际冷却（技能系统从 triggerParams 读取后执行）**
   ```
   actualCooldown = triggerParams["cooldown"] × (1 - cooldownReduction)
   ```
   - `cooldownReduction` 由"神速"等被动提供（0.0 - 0.8 范围，防止冷却归零）

3. **蓄力伤害系数（输入系统 + 技能系统协作）**
   ```
   chargeRatio = clamp(holdDuration / triggerParams["chargeTime"], 0, 1)
   ```
   - `holdDuration` 由输入系统提供
   - 仅 `OnClick` 且 `chargeTime > 0` 时执行此逻辑
   - 如何利用 `chargeRatio` 由技能系统自行定义（如线性伤害放大）

4. **自动触发间隔（OnTimer 触发器）**
   ```
   triggerInterval = triggerParams["interval"]
   ```

5. **升级权重抽取（升级系统执行）**
   ```
   P(skill) = skillConfig.weight / Σ(weight of all eligible skills)
   ```

## 边缘情况

1. **`skillId` 重复**
   - 情况：CSV 中两行拥有相同的 `skillId`
   - 处理：`SkillParser` 解析时检测重复 ID，后出现的记录覆盖前者并输出警告日志
   - 原因：`SkillDatabase` 为 `Dictionary<string, SkillConfig>`，重复键会静默覆盖，需显式告警

2. **`upgradesFrom` 引用不存在的 skillId**
   - 情况：某技能的 `upgradesFrom` 填写了一个 CSV 中不存在的 ID
   - 处理：`SkillParser` 完成全量加载后进行引用完整性校验，不存在的引用置为空字符串并输出警告
   - 原因：升级系统遍历进化链时会查询 `GetById()`，返回 `null` 会导致空引用异常

3. **`upgradesFrom` 循环引用**
   - 情况：技能 A 的 `upgradesFrom` 指向技能 B，技能 B 的 `upgradesFrom` 又指向技能 A
   - 处理：`SkillDatabase.GetUpgradesOf()` 遍历时设置最大深度上限（如 10），超出则截断并报错
   - 原因：升级系统遍历进化链时会无限循环

4. **`triggerParams` 或 `effectParams` 缺少必需参数**
   - 情况：`OnClick` 触发器没有提供 `cooldown` 参数
   - 处理：对应 Trigger / Effect 类在实例化时使用预定义默认值，并输出警告日志
   - 原因：防止参数缺失导致运行时异常，同时让数据错误可追溯

5. **`chargeTime` 参数为 0 但技能系统计算蓄力比例**
   - 情况：技能系统对 `chargeTime = 0` 的技能执行除法
   - 处理：技能系统在读取 `chargeTime` 后检查 `> 0`，为 0 时 `chargeRatio` 直接返回 1.0
   - 原因：防止除以零（`chargeRatio = holdDuration / chargeTime`）

6. **`weight = 0`**
   - 情况：设计师将某技能权重设为 0（永不出现）
   - 处理：升级系统跳过 `weight ≤ 0` 的技能，允许此行为（可用于临时禁用技能）
   - 原因：提供快速禁用技能的机制，无需删除 CSV 行

7. **`iconPath` 对应资源不存在**
   - 情况：`iconPath` 路径填写错误或图标尚未制作
   - 处理：UI 层加载失败时显示默认占位图标，不报错不崩溃
   - 原因：开发阶段图标尚未制作时，不应影响游戏功能

8. **CSV 文件格式错误（列数不符）**
   - 情况：某行列数与表头不匹配（如少了一列）
   - 处理：`SkillParser` 跳过该行并输出包含行号的警告日志，继续解析后续行
   - 原因：单行错误不应阻断整个数据库的加载

## 调优参数

所有战斗数值均存于 CSV 的 `triggerParams` / `effectParams` 列，修改后重启游戏即生效，无需重编译代码。推荐安全范围：

| 参数名 | 所属字段列 | 推荐范围 | 极端行为 |
|--------|----------|---------|---------|
| `damage`（effectParams） | effectParams | [0, 500] | > 500 可能单次点击秒杀精英单位，破坏节奏感 |
| `cooldown`（triggerParams） | triggerParams | [0.3, 5.0] | < 0.3s 视觉反馈堆叠混乱；> 5s 玩家感觉技能"不存在" |
| `chargeTime`（triggerParams） | triggerParams | [0, 3.0] | > 3s 蓄力过程太长，违反支柱 1 的节奏感 |
| `radius`（effectParams） | effectParams | [0, 8.0] | > 8 可能覆盖整个屏幕，使点击位置失去意义 |
| `interval`（triggerParams，OnTimer） | triggerParams | [1.0, 10.0] | < 1s 自动攻击频率过高，噪音感强；> 10s 存在感太低 |
| `weight` | level 列 | [0, 5.0] | > 5 可能使某技能几乎必然出现，破坏 Build 多样性 |
| `chance`（triggerParams，OnKill） | triggerParams | [0.0, 1.0] | 固定 1.0 表示必然触发；< 0.1 体感几乎不触发 |

## 验收标准

1. `SkillParser` 能正确解析标准格式的 `skills.csv`，不抛出异常
2. 解析完成后 `SkillDatabase.GetById("divine_finger")` 返回正确的 `SkillConfig` 对象
3. `triggerType` 和 `effectType` 字符串能通过反射正确实例化对应的 Trigger / Effect 类
4. `triggerParams` 和 `effectParams` 键值对能被正确拆分并供 Trigger / Effect 类读取
5. 重复 `skillId` 时输出警告日志，后出现的记录覆盖前者，不崩溃
6. `upgradesFrom` 引用不存在的 ID 时输出警告日志，置空，不崩溃
7. CSV 某行列数不符时跳过该行并输出行号警告，继续加载后续行
8. `GetStarterSkills()` 正确返回所有 `isStarterSkill = true` 的技能
9. 升级系统可正确读取 `weight` 字段生成加权随机升级选项
10. `iconPath` 对应资源不存在时，UI 显示占位图而不崩溃
11. 修改 CSV 数值后，游戏重新启动时生效（无需重编译代码）
