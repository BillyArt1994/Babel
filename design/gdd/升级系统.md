# 升级系统

> **状态：** 已完成
> **作者：** 独立开发者 + AI
> **最后更新：** 2026-03-28
> **实现支柱：** 支柱3（Build 策略深度）、支柱5（简洁的复杂性）

## 概述

升级系统管理玩家的成长循环：击杀敌人获得信仰值 → 信仰值达到阈值 → 游戏暂停进入升级选择 → 玩家 3 选 1 → 新技能加入技能系统 → 游戏继续。

本系统是支柱3（Build 策略深度）的核心载体。它从技能数据库读取权重生成选项池，不产生伤害也不执行逻辑，只负责"何时升级"和"提供什么选项"两件事。

## 玩家幻想

"我又快升级了……现在是选神雷锁链替换神罚之指，还是再叠一层神力被动？"——这种升级决策时刻是 Survivor-like 最核心的情感高峰。升级系统必须让每次 3 选 1 都感觉有意义：选项不重复、不提供无效选项（如已满层叠的被动），且偶尔出现稀有高权重选项时给玩家"中了"的惊喜感。

## 详细设计

### 核心数据结构

```csharp
public class UpgradeSystem : MonoBehaviour
{
    private float _faithAccumulated;    // 当前积累的信仰值
    private float _faithThreshold;     // 下次升级所需信仰值
    private int   _levelCount;         // 已触发的升级次数（从 0 开始）

    // 缓存全部技能资产（启动时从 Resources 加载）
    private SkillData[] _allSkills;

    // 当前待选的 3 个选项（生成后暂存，等待玩家选择）
    private SkillData[] _pendingOptions;
}
```

> ⚠️ **EnemyData 字段更新**：需在敌人数据库（#4）中新增 `faithValue: float` 字段。
> 参考数值：Worker = 1.0，Elite = 4.0，Priest = 2.0，Engineer = 2.0，Zealot = 1.0

### 核心规则

#### A. 信仰值积累

```csharp
// 订阅 EnemyEvents.OnEnemyDied
void OnEnemyDied(EnemyData data, Vector2 pos) {
    if (!GameLoopManager.IsPlaying()) return;  // 升级结算期间也不计入
    _faithAccumulated += data.faithValue;
    CheckUpgradeTrigger();
}
```

#### B. 升级触发检查

```csharp
void CheckUpgradeTrigger() {
    if (_faithAccumulated < _faithThreshold) return;

    // 保留溢出信仰值（下一级继续累积）
    _faithAccumulated -= _faithThreshold;

    // 更新下次阈值
    _faithThreshold = Mathf.Ceil(_faithThreshold * FAITH_SCALE_FACTOR);
    _levelCount++;

    // 生成 3 个选项
    _pendingOptions = GenerateOptions(3);

    // 触发游戏暂停 + 通知 UI
    GameLoopManager.RequestLevelUp();
    UpgradeEvents.OnOptionsGenerated?.Invoke(_pendingOptions);
}
```

#### C. 选项池生成

**资格判断（`IsEligible(SkillData skill)`）：**

| 条件 | 判断逻辑 |
|------|---------|
| 权重有效 | `skill.weight > 0` |
| ClickForm 资格 | `skill.skillType == ClickForm` 且 `skill != SkillSystem.GetActiveClickForm()`（不提供已有形态） |
| Passive 资格 | 当前叠加层数 `< skill.maxStackCount`（0 层 = 首次可选；达到上限则排除） |
| 进化链约束 | `skill.upgradesFrom == null` OR `SkillSystem.HasSkill(skill.upgradesFrom)` |

**加权随机选 3（不重复）：**

```csharp
SkillData[] GenerateOptions(int count) {
    var pool = _allSkills.Where(IsEligible).ToList();

    if (pool.Count == 0) return Array.Empty<SkillData>();
    count = Mathf.Min(count, pool.Count);  // 不足 3 个时全部展示

    var selected = new List<SkillData>();
    while (selected.Count < count && pool.Count > 0) {
        float totalWeight = pool.Sum(s => s.weight);
        float roll = Random.Range(0f, totalWeight);
        float cumulative = 0f;
        for (int i = 0; i < pool.Count; i++) {
            cumulative += pool[i].weight;
            if (roll < cumulative) {
                selected.Add(pool[i]);
                pool.RemoveAt(i);  // 无重复
                break;
            }
        }
    }
    return selected.ToArray();
}
```

#### D. 玩家选择后

```csharp
// 由升级选择 UI（#15）调用
public void SelectOption(int index) {
    if (_pendingOptions == null || index >= _pendingOptions.Length) return;

    SkillSystem.AddSkill(_pendingOptions[index]);
    _pendingOptions = null;

    GameLoopManager.RequestLevelUpComplete();
}
```

### 状态与转换

升级系统本身无状态机，但与 GameLoopManager 状态强耦合：

```
Playing 状态中击杀累积信仰值
    ↓ 达到阈值
RequestLevelUp() → GameState = LevelingUp（游戏暂停，UI弹出）
    ↓ 玩家选择
SelectOption() → GameState = Playing（游戏继续）
```

**信仰值溢出处理：**  
溢出信仰值**保留**到下一级（而非清空）。玩家在高效杀敌的情况下不会因为时序问题损失进度。

## 公式

**1. 升级阈值（对数增长）**
```
threshold[n] = ceil(FAITH_BASE × FAITH_SCALE_FACTOR ^ n)

n = 0（第 1 次升级）：ceil(10 × 1.2^0) = 10
n = 1（第 2 次升级）：ceil(10 × 1.2^1) = 12
n = 5（第 6 次升级）：ceil(10 × 1.2^5) = 25
n = 10（第 11 次升级）：ceil(10 × 1.2^10) = 62
n = 19（第 20 次升级）：ceil(10 × 1.2^19) = 313
```

**15 分钟内预期升级次数验证：**

| 阶段 | 约杀敌数 | 约平均信仰/杀 | 累计信仰 | 累计升级次数 |
|------|---------|------------|---------|------------|
| 0-5min | 300 | 1.1 | 330 | ≈ 8 次 |
| 5-10min | 500 | 1.5 | 750 | ≈ 13 次 |
| 10-15min | 700 | 1.8 | 1260 | ≈ 18-20 次 |

结论：约 18-20 次升级，平均每 45 秒一次，符合游戏概念的"30-60 秒升级一次"目标。

**2. 信仰值按血量比例**
```
faithValue = max(1.0, round(baseHealth / FAITH_HP_DIVISOR))
```
- `FAITH_HP_DIVISOR` = 30（Worker 的 baseHealth），使 Worker = 1.0 为基准
- 在 EnemyData 中手动配置，不自动计算

**3. 选项池加权概率**
```
P(skill) = skill.weight / Σ(weight of all eligible skills in pool)
```

## 边界情况

| 情况 | 处理方式 |
|------|---------|
| 可选池少于 3 个技能 | 展示全部可用选项（1 或 2 个），不填充占位 |
| 可选池为 0（所有技能已满）| 触发升级时 `GenerateOptions` 返回空数组，**跳过本次升级**（不弹 UI），信仰值重置为 0，阈值继续增长 |
| 同帧多个敌人死亡导致信仰值超过阈值的整数倍 | `CheckUpgradeTrigger` 每次调用只触发一次升级（溢出保留）；若需连续升级，下一帧的 EnemyDied 回调会再次触发 |
| 升级结算中玩家快速多次点击选项 | `SelectOption` 第一次调用后 `_pendingOptions = null`，后续调用立即返回 |
| 游戏在 LevelingUp 状态时又有敌人死亡（OnEnemyDied 仍在广播） | `OnEnemyDied` 内检查 `IsPlaying()`：`LevelingUp != Playing`，信仰值**不累积**（避免升级嵌套） |
| `FAITH_SCALE_FACTOR = 1.0`（不增长） | 每次升级需要相同信仰，会在 20 次升级后逼近对数曲线的平台期，可接受 |
| 升级时 `_faithAccumulated` 溢出多个阈值 | 仅处理一次，余量保留；下一次 `OnEnemyDied` 触发后再检查。不连锁升级，保留升级的仪式感 |

## 调节旋钮

| 旋钮 | 默认值 | 安全范围 | 极端行为 |
|------|--------|---------|---------|
| `FAITH_BASE` | 10 | 5 – 30 | < 5：前期太频繁升级，技能堆满太快；> 30：前 5 分钟几乎升不了级 |
| `FAITH_SCALE_FACTOR` | 1.2 | 1.0 – 1.5 | 1.0 = 固定阈值，后期升级太快；> 1.4 = 后期几乎停止升级 |
| `faithValue`（各敌人） | Worker=1, Elite=4, Priest=2, Engineer=2, Zealot=1 | [0.5, 10] 每类型 | Elite > 6：玩家专门猎精英忽视普通单位，破坏全场压力感 |

升级选择 UI（#15）需要的接口：
- 订阅 `UpgradeEvents.OnOptionsGenerated(SkillData[])` → 获取 3 张卡片数据

- 卡片描述：`description`  
- 卡片图标：`icon`

## 验收标准

1. **信仰积累**：击杀 Worker → +1 信仰；击杀 Elite → +4 信仰（数值准确）
2. **触发升级**：信仰达到阈值 → `RequestLevelUp()` 被调用，游戏暂停，3 选 1 界面出现
3. **溢出保留**：信仰值为 8，需 10 才升级，再杀 3 个 Worker → 信仰 = 1（溢出 1 保留）
4. **阈值增长**：连续触发 5 次升级，第 6 次阈值为 `ceil(10 × 1.2^5) = 25`
5. **选项去重**：同一次升级的 3 个选项互不重复
6. **当前形态不出现**：当前 ClickForm 为陨石，选项中不会再出现陨石
7. **满层被动排除**：神罚印记已达 maxStackCount，不再出现于选项池
8. **进化链约束**：若某技能 `upgradesFrom = X` 而玩家没有 X，该技能不出现
9. **池不足**：可选池只有 2 个技能时，展示 2 个选项（不崩溃）
10. **LevelingUp 中不累积信仰**：升级选择期间击杀敌人，信仰值不变化
11. **选择后继续**：`SelectOption()` 后 `RequestLevelUpComplete()` 被调用，游戏恢复 Playing

| # | 问题 | 优先级 | 备注 |

| 1 | 可选池为 0 时是否应提供"重新掷骰"（reroll）选项？ | 低 | MVP 直接跳过升级。若后期加 reroll，需要单独的货币系统。 |
